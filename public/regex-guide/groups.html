Gruppen & Lookarounds

Hier wird Regex erst richtig mächtig. Klammern teilen Muster ein. Lookarounds prüfen Kontext, ohne ihn zu verbrauchen. Alles Trickserei mit System.

Trick 1: Capturing Groups — die runde Klammer

Setz (...) um einen Teil deines Musters. Zwei Effekte:

  1. Die Gruppe wird als Ganzes von Quantoren behandelt: (ab)+ matcht „ab", „abab", „ababab".
  2. Was die Gruppe matcht, wird gespeichert und kann referenziert werden.
Muster:  (\d{4})-(\d{2})-(\d{2})
Text:    "2026-04-28"
Match:   2026-04-28
$1:      2026     ← erste Gruppe
$2:      04       ← zweite
$3:      28       ← dritte

Du kannst die Gruppen im selben Muster via \1, \2, ... wiederverwenden:

Muster:  (\w+) \1
Text:    "the the cat"   → Match auf „the the" (Wort, Space, dasselbe Wort)

Klassischer Einsatz: doppelte Wörter finden. Tabula rasa für Lektorinnen.

Trick 2: Non-Capturing — wenn du keine Beweise brinnen willst

Capturing kostet (wenig) Performance und vergibt Gruppennummern. Wenn du nur gruppieren willst, ohne den Inhalt zu speichern: (?:...).

(?:ab)+               gruppiert „ab", aber speichert nicht
(https?|ftp)          captured Protokoll
(?:https?|ftp)        gruppiert nur, du brauchst die Gruppe nicht weiter

Faustregel des Magiers: Default ist non-capturing. Capture nur, wenn du wirklich auf den Inhalt zugreifst.

Trick 3: Named Groups — Beschriftung statt Nummer

Statt $1, $2, $3 kannst du Gruppen benennen — selbsterklärend und stabiler bei Refactorings:

Muster:  (?<jahr>\d{4})-(?<monat>\d{2})-(?<tag>\d{2})
Text:    "2026-04-28"

In Code:
  match.group("jahr")   = "2026"
  match.group("monat")  = "04"

Notation variiert: (?<name>...) in PCRE/JS/.NET, (?P<name>...) in Python (Python akzeptiert beide ab 3.x).

Trick 4: Lookaround — der unsichtbare Zauber

Jetzt wird's elegant. Lookarounds prüfen, was vor oder nach der aktuellen Position steht — ohne diesen Text als Teil des Match zu zählen. Die heißen Zero-Width Assertions: sie verbrauchen keine Zeichen.

NotationBedeutung
(?=...)Lookahead positiv: nur Match, wenn ... folgt
(?!...)Lookahead negativ: nur Match, wenn ... NICHT folgt
(?<=...)Lookbehind positiv: nur Match, wenn ... vorausgeht
(?<!...)Lookbehind negativ: nur Match, wenn ... NICHT vorausgeht

Die Vorstellung

Muster:  \d+(?=€)
Text:    "Preis: 99€, Wert: 50$"
Match:   99       ← matcht Ziffern, ABER nur wenn ein € folgt
                    Das € selber wird NICHT mitmatched.
Muster:  (?<=\$)\d+
Text:    "Preis: 99€, Wert: 50$"
Wait... das matcht nichts, weil $ nicht VOR der 50 steht.

Korrigiert:
Muster:  \d+(?=\$)
Match:   50       ← Zahl, gefolgt von $

Negativer Lookahead in Aktion

Muster:  foo(?!bar)
Text:    "foobar foobaz fooqux"
Match:   foo in „foobaz" und in „fooqux"
         NICHT in „foobar" (weil bar folgt)

Lookbehind-Limitierung

Der Bühnenmagier muss eine Sache zugeben: Lookbehind ist in vielen Engines auf feste Länge beschränkt. (?<=foo) okay, (?<=fo+) nicht okay (variable Länge).

  • PCRE: feste Länge, später variabel
  • .NET: variable Länge erlaubt
  • Python regex-Modul: variable Länge
  • Python Standard re: feste Länge
  • JavaScript: ab Node 10 / moderne Browser variable Länge erlaubt
  • Go RE2: kein Lookbehind. Überhaupt nicht. Sorry.

Trick 5: Atomic Groups — die Falltür

Eine atomare Gruppe (?>...) ist wie eine non-capturing Gruppe, aber: einmal gematched, wird nie wieder zurückgegeben. Kein Backtracking innerhalb.

Der Effekt: schneller, deterministisch, immun gegen viele Catastrophic-Backtracking-Bugs. Mehr darüber jammert die Predigerin im nächsten Kapitel.
