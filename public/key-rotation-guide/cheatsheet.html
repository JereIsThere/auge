Standard-Playbook (6 Schritte)

  1. Inventur: wer benutzt diesen Key?
  2. Neuer Key generiert & im Secret-Manager.
  3. Beide Keys parallel akzeptiert (Overlap).
  4. Verbraucher migrieren.
  5. Audit: kein Use des alten Keys mehr?
  6. Alten Key disablen → 7 Tage Wartezeit → löschen.

Compromise-Trigger — sofort rotieren wenn:

  • Key in Git-History gefunden (auch in alten Commits!)
  • Key in Logs / Monitoring / Slack gepostet
  • Mitarbeiter mit Zugriff hat das Unternehmen verlassen
  • Ehemaliger Vendor / Auftragnehmer hatte Zugriff
  • Verdacht auf Phishing-Erfolg
  • Provider meldet Breach
  • Ungewöhnlicher Use im Audit-Log (anderer User-Agent, neue Geo, off-hours)
  • Geräte-Diebstahl (Laptop, Telefon mit Token-Apps)

Inventur-Befehle

# Git-History nach Secrets durchsuchen
trufflehog filesystem .
gitleaks detect --source .

# AWS: alle Access Keys + last used
aws iam list-users | jq -r '.Users[].UserName' | \
  xargs -I {} aws iam list-access-keys --user-name {}

# GitHub: Tokens deiner Org
gh api orgs/myorg/audit-log

# Kubernetes Secrets älter als 90 Tage
kubectl get secrets -A -o json | jq -r '.items[] | \
  select(.metadata.creationTimestamp < (now - 7776000 | todate)) | \
  "\(.metadata.namespace)/\(.metadata.name)"'

Audit-Checkliste (alle 6 Monate)

  • ☐ Alle aktiven Service-Account-Keys haben Owner + Purpose dokumentiert
  • ☐ Keys, deren Owner die Firma verlassen hat, sind revoked
  • ☐ Keys mit „last used" > 90 Tagen sind dokumentiert oder revoked
  • ☐ Cert-Ablaufdaten sind geMonitored, Reminders aktiv
  • ☐ KMS-Auto-Rotate ist aktiviert
  • ☐ Repository-Scans (truffleHog/gitleaks) haben keine Findings
  • ☐ Secret-Manager hat Audit-Log enabled
  • ☐ MFA für Provider-Konsolen erzwungen
  • ☐ Offboarding-Prozess testet Key-Revocation
  • ☐ TLS-Cert für jede öffentliche Domain < 30 Tagen vor Ablauf wird automatisch erneuert

Anti-Patterns (do NOT do)

❌ Hardcoded Keys im CodeUse Secrets-Manager
❌ Ein Key für alle ServicesPro Service einen Key
❌ Big-Bang-Rotation ohne OverlapOverlap, immer
❌ Key sofort löschen nach CutoverDisable + 7d Wartezeit
❌ Keys ohne AblaufdatumMaximal 1 Jahr, besser 90 Tage
❌ User-Password-Rotation nach ScheduleNur on suspicion (NIST 2017+)
❌ Key-Rotation ohne Audit-LogLogge Create / Use / Revoke
❌ Mobile-App mit hardcoded Public KeyJWKS mit Refresh-Mechanismus

One-Liner-Mantra

„Generate new. Run both. Migrate consumers. Verify silence on old. Disable. Wait. Delete."
