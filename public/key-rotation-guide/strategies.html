Rotation-Strategien

„Neuen Key generieren, alten löschen" funktioniert nur in einem leeren Raum. In Production brauchst du Strategien, die Downtime ausschließen. Hier sind die fünf, die wirklich funktionieren.

Strategie 1: Overlap (auch „Rolling Rotation")

Der Klassiker. Beide Keys sind eine Zeit lang gültig.

  1. Neuen Key generieren. Alten weiterhin akzeptieren.
  2. Neuen Key in alle Verbraucher ausrollen (Apps, CI, Cron-Jobs, Mobile-Clients).
  3. Warten. Nicht zwei Stunden. Tage. Bis du sicher bist, dass kein Verbraucher mehr den alten benutzt — Audit-Logs prüfen.
  4. Alten Key revoken.

Fast jedes vernünftige System unterstützt Overlap: AWS erlaubt 2 Access-Keys pro User, JWT-Validation kann mehrere Public-Keys checken, OAuth-Apps haben Client-Secrets-Versionierung. Wenn dein System Overlap nicht unterstützt, ist das ein Architektur-Bug, kein Operational-Problem.

Strategie 2: Versioning (Key-IDs)

Statt „der Key" gibt es „Key v1", „Key v2", ... Tokens werden mit Version signiert (z.B. JWT kid-Header). Verifier kennen alle aktiven Versionen.

Vorteil: Du kannst alte Keys weiter verifizieren, ohne neue damit auszustellen. Token-Lebensdauer entkoppelt von Key-Wechsel.

JWT-Header: { "alg": "RS256", "kid": "v3" }

Verifier holt Key v3 aus seinem Keystore.
Token mit kid="v2" wird noch akzeptiert (alt, aber gültig).
Token mit kid="v1" wird abgelehnt (zurückgezogen).
Neue Tokens werden mit v4 signiert.

Strategie 3: Phasing — gestaffeltes Rollout

Großes System, viele Verbraucher? Rotation in Wellen:

  1. Phase 1: Neuer Key generiert. Bei 1% des Traffics einsetzen (Canary).
  2. Phase 2: 24h beobachten. Errors? Rollback.
  3. Phase 3: 10%, 50%, 100% in Schritten.
  4. Phase 4: Alten Key revoken nach voller Migration.

Klingt nach Overkill für einen API-Key. Ist es nicht, wenn der Key in 200 Microservices verwendet wird und nicht alle deine Doku gelesen haben.

Strategie 4: Automatische Rotation

Mensch ist die unzuverlässigste Komponente. Automatisierung tut das Richtige zum richtigen Zeitpunkt:

  • Cloud-KMS rotiert Crypto-Keys jährlich automatisch. Für jede neue Verschlüsselung wird neuer Key benutzt; Entschlüsselung alter Daten geht weiter.
  • cert-manager erneuert TLS-Certs in Kubernetes 30 Tage vor Ablauf.
  • AWS Secrets Manager kann DB-Credentials rotieren — schreibt das Passwort selbst, deployed an alle Verbraucher.
  • HashiCorp Vault mit dynamischen Secrets — DB-User wird pro Session generiert, lebt 1h, wird dann gelöscht.

Das ist der Endgame. Wer manuell rotiert, weiß, dass er manuell vergessen wird.

Strategie 5: Re-Encrypt für Encryption-Keys

Bei DEKs (Data Encryption Keys) ist „Rotation" trickier — die Daten sind verschlüsselt damit. Optionen:

  • Lazy Rewrap: alte Daten bleiben mit altem Key. Neue Daten kriegen neuen Key. Beim nächsten Schreiben werden alte Daten umverschlüsselt.
  • Eager Re-Encrypt: Background-Job rewrapt alle Daten mit neuem Key.
  • Envelope-Encryption: jede Datei hat ihren eigenen DEK, der mit einem KEK verschlüsselt ist. Bei Rotation muss nur der KEK rotieren — alle DEKs werden neu wrapped, die Daten bleiben unangerührt. Standard-Pattern

Mein 6-Schritte-Standard-Playbook

  1. Inventur: wo ist der Key, wer benutzt ihn?
  2. Neuer Key generiert & im Secret-Manager hinterlegt.
  3. Beide Keys parallel akzeptiert (Overlap-Mode).
  4. Verbraucher migrieren — Config-Update, Pod-Restart, Cron-Reload.
  5. Verifikations-Phase: Audit-Logs prüfen, kein Use des alten Keys mehr.
  6. Alten Key deaktivieren (nicht löschen, nur disablen — falls etwas auftaucht).
  7. Nach 7 Tagen ohne Vorfall: alten Key endgültig löschen.

Schritt 1 wird unterschätzt. „Wo wird dieser Key verwendet?" beantwortet niemand auf Anhieb. Inventarisierung ist die meiste Arbeit. Tools: git grep, Provider-Audit-Logs, Secret-Manager-Access-Logs.
