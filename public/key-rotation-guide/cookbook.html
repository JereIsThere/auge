Cookbook

Konkrete Rotation-Anleitungen pro Key-Typ. Schritte, Befehle, Stolperfallen. Hand-zertifiziert.

Rezept 1: AWS Access Keys

# 1. Neuen Key erstellen (nur ein User-Key, neuer parallel)
aws iam create-access-key --user-name myuser

# 2. Output enthält neuen AccessKeyId + SecretAccessKey
#    → in Secrets Manager / Vault hinterlegen

# 3. Alle Verbraucher (CI, Apps, Cron) auf neuen Key umstellen.

# 4. Alten Key deaktivieren (NICHT löschen!)
aws iam update-access-key --access-key-id AKIA_OLD --status Inactive

# 5. 7 Tage warten. Wenn Audit-Logs sauber:
aws iam delete-access-key --access-key-id AKIA_OLD

Pro User maximal 2 Access-Keys. Wenn du das 3. brauchst, hast du falsch organisiert.

Rezept 2: OpenAI / Anthropic API-Keys

  1. Im Provider-Dashboard neuen Key erstellen, mit aussagekräftigem Namen (prod-2026-Q2).
  2. In Secrets-Manager hinterlegen.
  3. Apps auf neuen Key umstellen — Rolling Restart deiner Pods.
  4. Im Provider-Dashboard alten Key revoken.

Trickiges Detail: viele dieser APIs erlauben kein Overlap auf Service-Ebene — beide Keys sind aktiv, bis du alten revokest. Plan für kurzes Zeitfenster, in dem dein Code beide Werte aus Config liest.

Rezept 3: GitHub Personal Access Token / Fine-Grained Token

  1. Settings → Developer Settings → Tokens → neuen erstellen, Scopes minimal halten.
  2. In CI-Secrets, lokalen Tools, gh-CLI ersetzen.
  3. Alten widerrufen.

Bonus: Fine-grained tokens haben Ablaufdatum mit Maximum 1 Jahr. Klassische PATs erlauben „No expiration" — bitte nie. Setz immer ein Ablauf.

Rezept 4: SSH-Keys

# 1. Neuen Key generieren (modern: ED25519)
ssh-keygen -t ed25519 -C "max@laptop-2026" -f ~/.ssh/id_ed25519_new

# 2. Public Key auf alle Server, GitHub, GitLab pushen
ssh-copy-id -i ~/.ssh/id_ed25519_new.pub user@server1
gh ssh-key add ~/.ssh/id_ed25519_new.pub --title "laptop-2026"

# 3. SSH-Config anpassen, sodass neuer Key benutzt wird
# ~/.ssh/config:
Host *
  IdentityFile ~/.ssh/id_ed25519_new
  IdentitiesOnly yes

# 4. Test: Login auf alle Targets, ob neuer Key funktioniert
ssh user@server1 "echo OK"

# 5. Alten Public Key bei GitHub/GitLab/Servern entfernen
# auf Servern: aus authorized_keys-Datei rauseditieren

# 6. Lokal alten Key sicher löschen
shred -u ~/.ssh/id_ed25519_old

Rezept 5: TLS-Zertifikate (Let's Encrypt)

Auto-Renewal sollte schon laufen. Falls nicht:

sudo certbot renew --dry-run    # testet, ob Renewal funktioniert
sudo certbot renew              # echtes Renewal
# Service reloaden: nginx -s reload, oder via certbot --post-hook

Faustregel: certbot erneuert 30 Tage vor Ablauf. Wenn das nicht in cron oder systemd-timer steht, ist dein Setup kaputt — fix das, bevor du irgendwas anderes machst.

Rezept 6: JWT-Signing-Keys (RS256)

  1. Neues Schlüsselpaar generieren mit eindeutigem kid (z.B. v4).
  2. Beide Public-Keys in deinem JWKS-Endpoint exponieren — https://yoursite/.well-known/jwks.json liefert v3 + v4.
  3. Issuer (Auth-Server) signiert ab sofort mit v4.
  4. Alle alten Tokens (mit v3) laufen normal aus — durch ihre eigene Lebensdauer.
  5. Nach Ablauf der maximalen Token-TTL: v3 aus JWKS entfernen.
# Neuen RSA-Key generieren
openssl genrsa -out jwt_v4.pem 4096
openssl rsa -in jwt_v4.pem -pubout -out jwt_v4_pub.pem

Rezept 7: Datenbank-Passwörter

Direkter Weg ohne Downtime:

-- 1. Zweiten User mit gleichen Permissions
CREATE USER app_v2 WITH PASSWORD '...neuesPasswort...';
GRANT ALL ON DATABASE myapp TO app_v2;

-- 2. App auf app_v2 umstellen, deployen.

-- 3. Wenn alle Pods neu laufen:
DROP USER app;

Wer auf einem User das Passwort wechselt: alle aktiven Connections kriegen Auth-Errors beim Reconnect. Bei Pooling = Outage. Mit zwei Usern + Cutover sauberer.

Rezept 8: Webhook-Signing-Secret (HMAC)

  1. Sender unterstützt typischerweise einen „secondary secret"-Slot. Stripe, GitHub, Shopify, Slack — alle haben das.
  2. Neues Secret im Provider-Dashboard hinzufügen.
  3. Receiver-Code: prüft Signatur gegen beide Secrets, akzeptiert wenn eines matcht.
  4. Im Provider altes Secret revoken.
  5. Receiver-Code: zweite Prüfung wieder rausnehmen (oder lassen, wenn dein Code sauber ist).

Rezept 9: Encryption-Key (KEK in KMS)

Cloud-KMS (AWS, GCP, Azure) hat automatische jährliche Rotation. Aktivieren:

# AWS
aws kms enable-key-rotation --key-id alias/mykey

# Resultat: jährlich automatisch neue Key-Version. Alte Versionen
# bleiben für Entschlüsselung verfügbar. Du machst gar nichts.

Außerhalb-KMS: Envelope-Encryption-Setup ist Pflicht, sonst musst du alle Daten re-encrypten. Das ist eigene Architektur-Entscheidung, kein Rotation-Pattern.

Rezept 10: 1Password / Passwort-Manager-Items

Wenn du in 1Password ein Item rotieren willst:

  1. Item in 1Password öffnen.
  2. Beim Provider neues Passwort generieren (1Password generiert dir eins, wenn du auf das Recipe-Symbol klickst).
  3. Beim Provider altes durch neues ersetzen (oder zwei aktiv haben für Overlap, wenn Provider das erlaubt).
  4. Im 1Password-Item unter „Password History" siehst du die alten Passwörter — gut für Audit.

Rezept 11: SSH-Server-Host-Keys

Selten nötig — nur, wenn ein Server kompromittiert war. Sonst nicht anfassen.

sudo ssh-keygen -A   # neue Host-Keys generieren
sudo systemctl restart sshd

# Alle Clients kriegen jetzt „Host key has changed!"-Warning.
# Du musst sie informieren und den neuen Fingerprint bekanntgeben.

Daher: Host-Keys nie unnötig rotieren. Wenn Server kompromittiert: ja. Sonst: lass.

Rezept 12: npm-publish-Token

  1. npmjs.com → Account → Access Tokens → Generate New Token (Granular, mit nur den Packages, die du publishest).
  2. In CI-Secret NPM_TOKEN ersetzen.
  3. Alten Token revoken.

npm-Tokens sollten nie in .npmrc-Files in Repos liegen. Auch nicht in privaten. Wenn ein Repo geleakt wird, geht der Schaden über das Repo hinaus — Angreifer kann böswillige Versionen deiner Packages publishen.

Hier ist der Lappen. Hände waschen vor dem Mittagessen. Nächste Station: was passiert, wenn was schief geht.
