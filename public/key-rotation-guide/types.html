Welche Keys gibt es eigentlich?

Bevor wir rotieren, müssen wir kategorisieren. Verschiedene Key-Arten haben verschiedene Bedrohungsmodelle, Lebenszyklen und Werkzeuge. Hier ist die Taxonomie.

Klasse 1: Symmetrische Geheimnisse („Shared Secrets")

Ein Wert, den beide Seiten kennen. Wer ihn hat, kann sich authentisieren oder Daten entschlüsseln.

  • API-Keys (OpenAI, Stripe, AWS, GitHub Personal Access Tokens). Lange Strings, oft mit Prefix wie sk-..., ghp_....
  • Webhook-Secrets (HMAC-Signatur-Schlüssel)
  • Datenbank-Passwörter
  • Encryption-Keys (DEK/KEK) — der Schlüssel, mit dem Daten at rest verschlüsselt werden
  • JWT-Signing-Secrets (HS256)
  • Session-Cookies-Secret

Charakteristika: leicht zu kopieren, leicht zu verlieren. Höchste Rotations-Priorität.

Klasse 2: Asymmetrische Schlüsselpaare

Public + Private. Privat wird gehütet, public verteilt.

  • SSH-Keys — User-Login, Host-Identifizierung
  • TLS-Zertifikate — HTTPS, mTLS
  • JWT-Signing-Keys (RS256, ES256) — eines unterschreibt, viele verifizieren
  • GPG-Keys — Mail-Signaturen, Commit-Signaturen, Package-Signing
  • SSH-Host-Keys — Identität deines Servers für Clients

Charakteristika: Private bleibt lokal, Public muss verteilt werden. Rotation = neuer Public-Key bei allen Trustern eintragen.

Klasse 3: Ephemere / Session-Tokens

  • OAuth Access-Tokens (kurz: 15min–1h)
  • OAuth Refresh-Tokens (länger: Tage–Monate)
  • Browser-Session-Cookies
  • STS-Tokens (AWS, GCP)

Charakteristika: kurzlebig by design. „Rotation" passiert automatisch durch Refresh oder Re-Login. Fokus liegt darauf, dass echtes Refresh implementiert ist und nicht „1 Jahr Token, klingt sicher".

Klasse 4: Provider-managed

Du klickst auf „Rotate" — der Provider macht den Rest. Beispiele:

  • AWS IAM Access Keys — pro User max. 2 aktive, sodass Rollover möglich
  • Cloud-KMS-Keys (AWS KMS, GCP KMS, Azure Key Vault) — automatische jährliche Rotation auf Knopfdruck
  • Database-managed credentials (AWS RDS Secrets Manager, GCP Cloud SQL)
  • Service-Account-Keys (GCP)

Charakteristika: am einfachsten zu rotieren. Sollte dein Default-Choice sein, wo verfügbar.

Klasse 5: Hardware-gebundene Keys

  • YubiKeys / FIDO2 — Private Key verlässt nie das Device
  • HSMs — Hardware-Security-Modules in Rechenzentren
  • TPM-gespeicherte Keys
  • Mobile-Secure-Enclave (Apple, Android Keystore)

Charakteristika: Private existiert nur auf dem Device. Rotation = neues Device pairen, altes deauthorizen.

Klasse 6: Passwörter (User-controlled)

Streng genommen Klasse-1, aber separat behandelt:

  • User-Passwörter
  • Master-Passwort eines Passwort-Managers
  • Recovery-Codes

Charakteristika: nicht zu oft rotieren! Studien zeigen: erzwungene 90-Tage-Rotation führt zu schwächeren Passwörtern (User hängen +1 dran). NIST empfiehlt seit 2017: nur bei Verdacht auf Compromise.

Klasse 7: Kryptografisches Material in Anwendungen

  • App-Signing-Keys (Android Play, iOS-Distribution)
  • Code-Signing-Certs (Microsoft Authenticode, Apple Developer)
  • Package-Signing (apt, rpm, npm-Provenance)

Charakteristika: super-langlebig. Wenn Android-Signing-Key leakt, sind alle Updates für die App kompromittiert. Rotation kostet User-Action (App neu installieren). Hochwert-Asset.
