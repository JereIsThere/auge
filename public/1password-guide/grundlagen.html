Grundlagen — Items richtig anlegen

Der Unterschied zwischen einem chaotischen 1Password und einem, das dir wirklich Arbeit abnimmt, liegt in der Item-Hygiene. Hier ist, wie du es von Anfang an sauber machst.

1. Die Item-Typen — und welche du wirklich brauchst

1Password hat ~20 Item-Typen. Die meisten Leute brauchen nur fünf bis sieben. Verwende den richtigen Typ — er bestimmt, welche Felder du bekommst, wie Autofill funktioniert und wie die Daten in der Suche auftauchen.

TypWofürWichtig
LoginJeder Account mit Username/PasswortDefault-Wahl. Mit URL-Feld → Autofill.
PasswordNur ein Passwort, kein Account drumrumWLAN-Schlüssel, alte Notizen. Selten nötig.
Secure NoteFreitext mit MarkdownRecovery Codes, Notfall-Infos, geheime Dokus.
Credit CardKartenAutofill in Checkout-Formularen.
IdentityAdresse + Telefon + E-MailAutofill für Versand-/Registrierungs-Forms.
API CredentialAPI-Keys, TokensFelder für Type, Filename, Valid From/To. Dev
SSH KeySSH-SchlüsselGeneriert direkt in 1Password, nutzt SSH-Agent. Dev
DatabaseDB-ConnectionsHost, Port, User, Pass, DB-Name strukturiert.
ServerRoot-/SSH-LoginsAdmin-Console + SSH zusammen.
Software LicenseLizenzkeysVersionsnummer + Key + Email + Bestelldatum.
DocumentDatei-AnhängePDFs, Scans (Pässe, Verträge).
Bank Account / Passport / Driver LicenseStrukturierte PersönlichesKlare Felder > Freitext. Suche findet's besser.
Faustregel Wenn ein dedizierter Item-Typ existiert, nimm ihn. Eine Kreditkarte als „Login" anzulegen funktioniert technisch — aber Autofill, Suche und 1Password-Watchtower behandeln sie dann nicht richtig.

2. Login-Felder richtig benutzen

Ein Login hat fünf Standard-Felder. Sie haben jeweils eine spezifische Funktion — füll sie nicht „kreativ" um.

Username

Der eigentliche Login-Identifier. Bei den meisten Diensten ist das die E-Mail, manchmal ein Username. Trag nichts anderes hier rein (kein „John Doe", kein „mein Hauptaccount") — Autofill nimmt den Inhalt 1:1.

Password

Generiert über das eingebaute Password-Recipe (Würfelsymbol). Default: 20 Zeichen, Letters + Numbers + Symbols.

💡 Pro-Move Pro Login das Recipe anpassen: manche Banken erlauben max. 16 Zeichen oder keine Sonderzeichen. Klick auf das Recipe-Symbol → "Customize" → speichern. 1Password merkt sich das per Item.

Website (URL)

Das wichtigste Feld für Autofill. Format-Tipp: Nur die Domain, ohne Pfad, ohne Trailing-Slash:

✅ https://example.com
✅ https://app.example.com
❌ https://example.com/login?ref=xyz
❌ example.com

One-Time Password (TOTP)

Siehe unten. Pflicht für jedes Konto, das 2FA anbietet.

Notes

Markdown wird gerendert. Nutze es für: Recovery-Hints, „welche Mailadresse hat sich registriert", Hotline-Nummern, Sicherheitsfragen-Antworten.

Rich Icons (automatische Logos)

1Password holt automatisch das Favicon der Domain im URL-Feld. Damit es funktioniert:

  • Setting an: Settings → Appearance → "Show icons" (Desktop) bzw. "Show website icons" in der Browser-Extension.
  • URL muss mit https:// beginnen und sollte nur die Domain enthalten — keine Pfade, keine Query-Strings.
  • Nach Speichern lädt das Icon ein paar Sekunden später nach.

Falsches Logo? Item bearbeiten → klick auf das aktuelle Icon (oben links beim Titel) → "Choose an icon" → eigenes Bild hochladen oder anderen Vorschlag wählen. Custom-Icons überschreiben das Auto-Favicon dauerhaft.

Trick: hat eine Site mehrere Domains, häng als zusätzliche URL die mit dem schöneren Favicon dran (z.B. https://github.com zusätzlich zu https://api.github.com) — 1Password nimmt typischerweise die erste mit Erfolg-Favicon.

3. Mehrere URLs pro Login — der unterschätzte Move

Ein Login kann beliebig viele URLs haben. Ein Item, viele Domains. Klick im URL-Feld auf "+ add another website".

Wann du das brauchst:

  • google.com, youtube.com, gmail.com → ein einziger Google-Login
  • github.com + github.dev + codespaces.new
  • Firma migriert von oldcorp.com auf newcorp.io → beide eintragen, kein Duplikat
  • SaaS-Tools mit Subdomains: acme.atlassian.net, acme.slack.com

URL-Matching-Modi

Pro URL kannst du den Match-Modus setzen (Klick auf das Drop-Down neben der URL):

Base domainDefault. example.com matcht auch app.example.com.
HostExakter Hostname inkl. Subdomain. Für getrennte Logins auf Subdomains.
ExactURL muss exakt matchen — Pfad inklusive. Selten nötig.
NeverNie autofillen. Für „diese URL gehört dazu, aber nicht autofillen".

4. TOTP (2FA) direkt in 1Password

Du kannst Authy / Google Authenticator komplett rauswerfen. 1Password generiert TOTP-Codes und trägt sie automatisch ein.

So fügst du einen Code hinzu

  1. Bei Setup-2FA-Seite des Dienstes auf „QR-Code" klicken.
  2. In 1Password → Login öffnen → Edit → "+ add field" → "One-Time Password".
  3. Im Feld auf das QR-Symbol klicken → 1Password scannt den Code vom Bildschirm. Alternativ: Secret-String reinpasten (steht meistens unter dem QR-Code als „can't scan?").
  4. Backup-Codes als Secure Note oder als Custom Field „Recovery Codes" beim selben Item.
⚠️ Die alte Diskussion: TOTP im selben Tool wie das Passwort? Klassisch sagt man: 2FA = zweiter Faktor = sollte separat sein. Praktisch: 1Password ist mit Master-Passwort + Secret Key + (optional) Hardware-Key bereits stark genug abgesichert, dass der „compromised vault → alles weg"-Fall extrem unwahrscheinlich ist. Für die meisten Leute überwiegt der Komfort. Für kritische Accounts (E-Mail-Hauptaccount, 1Password selbst, Bank): zweiter Faktor mit Hardware-Key (YubiKey) statt TOTP.

5. Custom Fields & Sections

Jedes Item kann beliebig viele eigene Felder bekommen. Im Edit-Modus → "+ add more" → "Section". Innerhalb der Section: "+ add field".

Feld-Typen — und wann du welchen nimmst

TextSichtbar, kopierbar, in der Suche
PasswordVerdeckt, kopierbar, nicht in der Suche
One-Time PasswordGeneriert TOTP-Codes
Date / Month-YearFür Ablaufdaten, Geburtstage
Credit Card Number / PINMit speziellem Autofill-Verhalten
URLAnklickbar, aber kein Autofill-Trigger (das macht nur das Haupt-URL-Feld)
Email / PhoneStrukturiert für Kontaktdaten
FileAnhang (Image, PDF, beliebig)

Beispiel: ein gut strukturierter GitHub-Login

Title: GitHub
Username: max@example.com
Password: ●●●●●●●●●●●●●●
Website: https://github.com (Base domain)
Website: https://github.dev (Base domain)
TOTP: ●●●●●●

Section "Recovery"
  recovery-codes: a1b2-c3d4 ... (Password field, multi-line)
  email-on-file: max@example.com

Section "API"
  personal-access-token: ghp_xxx (Password field)
  token-scopes: repo, workflow, write:packages (Text)
  token-expires: 2026-12-31 (Date)

Tags: dev, github

6. Tags vs. Vaults vs. Categories

Drei Konzepte, die ständig verwechselt werden. Klare Trennung:

WasWofür
VaultContainer mit ZugriffsrechtenTrennung nach wer darf rein: Privat, Familie, Arbeit.
CategoryItem-Typ (Login, Note, ...)Wird automatisch durch den Typ gesetzt.
TagBeliebige Labels, mehrere pro ItemQuerschnitts-Sortierung: finance, dev, 2026-cleanup, kids.

Tags sind hierarchisch: work/clients/acme erzeugt verschachtelte Tags. In der Sidebar sieht das aus wie ein Ordnerbaum. Verschachteln ist mächtig — aber ab Tiefe 3 wird's unübersichtlich.

7. Item-Hygiene-Checkliste

Geh diese Liste einmal pro Quartal durch — dauert eine halbe Stunde, hält den Vault sauber:

  • ☐ Watchtower → "Compromised Passwords" leer? Sonst rotieren.
  • ☐ Watchtower → "Reused Passwords" — alle bis auf bewusste Ausnahmen rotieren.
  • ☐ Watchtower → "Weak Passwords" auf neue Mindeststärke bringen.
  • ☐ "Inactive 2FA" — Sites, die 2FA anbieten, aber bei dir nicht aktiviert ist.
  • ☐ Items ohne Tags → mindestens einen Tag vergeben.
  • ☐ Doppelte Logins (Watchtower → "Duplicate Items") zusammenführen oder einen löschen.
  • ☐ Tote Accounts (kündigen) löschen — nicht nur das 1Password-Item entfernen, sondern den Account beim Anbieter.
