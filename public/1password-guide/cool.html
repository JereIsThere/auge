Neat Shit — CLI, SSH, Secrets

Hier wird's geil. 1Password kann .env-Dateien ohne Plaintext-Secrets, deine SSH-Keys verwalten und mit einem Touch deine Git-Commits signieren. Wenn du Dev bist, ist das der Teil, der dein Leben verändert.

1. 1Password CLI installieren & verbinden

# macOS
brew install --cask 1password-cli

# Linux (Debian/Ubuntu)
curl -sS https://downloads.1password.com/linux/keys/1password.asc | \
  sudo gpg --dearmor --output /usr/share/keyrings/1password-archive-keyring.gpg
echo "deb [arch=$(dpkg --print-architecture) signed-by=/usr/share/keyrings/1password-archive-keyring.gpg] \
  https://downloads.1password.com/linux/debian/$(dpkg --print-architecture) stable main" | \
  sudo tee /etc/apt/sources.list.d/1password.list
sudo apt update && sudo apt install 1password-cli

# Windows
winget install AgileBits.1Password.CLI

In der Desktop-App: Settings → Developer → ✅ "Integrate with 1Password CLI" + ✅ "Biometric unlock for 1Password CLI". Das verbindet op mit der App, sodass du nicht jedesmal das Master-Passwort tippen musst — Touch ID / Windows Hello reicht.

op signin            # einmal pro Shell-Session
op vault list        # checkt, dass alles läuft
op item list --vault Private

2. Secret References — die Killer-Idee

Statt Secrets in .env, in CI-Variablen oder im Code zu speichern, schreibst du einen Reference-String:

op://<Vault>/<Item>/<Field>

Beispiele:

op://Private/GitHub/credential
op://Work/AWS Production/access_key_id
op://Work/Stripe/api_key
op://Private/SSH GitHub/private_key

Diese Strings sind nicht geheim. Du kannst sie ins Repo committen, in Slack posten, in Dotfiles ablegen. Der eigentliche Wert wird erst zur Laufzeit aus 1Password aufgelöst — und nur, wenn dein Vault entsperrt ist.

Reference holen ist einfach: in der Desktop-App auf einem Feld Rechtsklick → "Copy Secret Reference".

op run — env vars ohne plaintext .env

Statt .env mit echten Secrets:

# .env (committen!)
DATABASE_URL=op://Work/Prod DB/connection_string
STRIPE_KEY=op://Work/Stripe/api_key
GITHUB_TOKEN=op://Private/GitHub/credential

Programm starten:

op run --env-file=.env -- npm start
op run --env-file=.env -- python app.py
op run --env-file=.env -- ./run-deploy.sh

1Password löst die Refs nur im Speicher des Subprozesses auf. Auf der Platte liegen niemals Plaintext-Secrets.

💡 Killer-Move Schreib in .gitignore NUR .env.local. .env mit den Refs darf ins Repo. Onboarding eines neuen Devs: git clone → 1Password installieren → op signin → fertig. Keine Secrets per Slack mehr.

op inject — Templates rendern

Für Configs, die kein .env-Format haben (z.B. Kubernetes-Secrets, JSON-Configs, Nginx-Configs):

# config.template.yaml
database:
  host: db.internal
  password: op://Work/Prod DB/password
api:
  stripe_key: op://Work/Stripe/api_key
op inject -i config.template.yaml -o config.yaml

Erzeugt eine fertige config.yaml mit den aufgelösten Werten. Pipe-fähig:

op inject -i k8s-secret.template.yaml | kubectl apply -f -

5. SSH-Agent — Keys mit biometrischem Touch

1Password kann ein voller SSH-Agent sein. Deine Private-Keys liegen verschlüsselt in 1Password. Jede SSH-Auth braucht einen biometrischen Touch.

Setup

  1. Settings → Developer → ✅ "Use the SSH agent".
  2. Existing Key importieren: in 1Password "+ New Item" → "SSH Key" → "Import" → vorhandene id_ed25519 hochladen.
    Oder neu generieren: "SSH Key" → "Generate" → ED25519.
  3. Public Key kopieren → bei GitHub/GitLab/Server hinzufügen.
  4. Setze in ~/.ssh/config: 
    Host *
  IdentityAgent ~/Library/Group\ Containers/2BUA8C4S2C.com.1password/t/agent.sock
  # Linux: ~/.1password/agent.sock
  # Windows: \\.\pipe\openssh-ssh-agent

Jetzt: ssh user@host → 1Password popup → Touch ID → drin. Der Private-Key verlässt 1Password nie.

Mehrere Keys, automatische Auswahl

1Password schickt automatisch alle gespeicherten Keys mit. Server pickt den passenden. Wenn du Kontrolle willst:

Host github.com
  IdentitiesOnly yes
  IdentityFile ~/.ssh/id_personal.pub   # Pub-Key zur Auswahl

Host gitlab.work.internal
  IdentitiesOnly yes
  IdentityFile ~/.ssh/id_work.pub

6. Git-Commits mit SSH-Key signieren

Statt GPG (Schmerz): SSH-Key zum Signieren. Mit 1Password = jeder Commit braucht Touch ID.

git config --global gpg.format ssh
git config --global user.signingkey "ssh-ed25519 AAAA... your_email@..."
git config --global commit.gpgsign true
git config --global gpg.ssh.program \
  "/Applications/1Password.app/Contents/MacOS/op-ssh-sign"

Bei GitHub: Settings → SSH and GPG keys → "Signing Key" (separat von Authentication-Key, aber kann derselbe sein).

git commit → Touch ID → Commit zeigt grünes "Verified" Badge auf GitHub.

7. CLI-Plugins

1Password kann sich vor andere CLI-Tools schalten und Auth transparent injizieren. Plugin einrichten:

op plugin init gh         # GitHub CLI
op plugin init aws        # AWS CLI
op plugin init stripe     # Stripe CLI
op plugin init cargo      # crates.io
op plugin init databricks
# ... ~30 Plugins verfügbar

Danach läuft gh repo list — und 1Password injiziert deinen GitHub-Token aus dem Vault, ohne dass er irgendwo auf der Platte liegt. gh auth login brauchst du nie wieder.

8. Service Accounts für CI/CD

In GitHub-Actions / GitLab-CI willst du Secrets aus 1Password ziehen — aber kein menschlicher Account-Login.

Lösung: Service Account. Eigener Token, scoped auf bestimmte Vaults, read-only.

  1. my.1password.com → Developer → Service Accounts → New.
  2. Vaults wählen, Permissions setzen (meist read-only).
  3. Token wird einmal angezeigt — kopieren, in GitHub-Secret OP_SERVICE_ACCOUNT_TOKEN ablegen.

In der GitHub Action:

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: 1password/load-secrets-action@v2
        with:
          export-env: true
        env:
          OP_SERVICE_ACCOUNT_TOKEN: ${{ secrets.OP_SERVICE_ACCOUNT_TOKEN }}
          STRIPE_KEY: op://Work/Stripe/api_key
          DATABASE_URL: op://Work/Prod DB/connection_string
      - run: ./deploy.sh   # Secrets sind als env vars verfügbar

Vorteil: keine Secrets in GitHub-Secrets selbst — nur das eine Service-Account-Token. Rotation passiert in 1Password, GitHub merkt nichts.

9. 1Password for Developers in IDEs

  • VS Code Extension "1Password for VS Code" — markiert Secrets im Code, bietet "Save in 1Password" an, autocomplete für Secret-Refs.
  • JetBrains Plugin — analog für IntelliJ/PyCharm/WebStorm.
  • Pre-commit-Hook: op kann scannen, ob versehentlich Plaintext-Secrets in einen Commit rutschen.
# .pre-commit-config.yaml
repos:
  - repo: https://github.com/1Password/op-pre-commit
    rev: v1.0.0
    hooks:
      - id: detect-secrets
