Eigener Server (hbbs/hbbr)

Der wahre Grund, RustDesk zu nutzen. Auf einem 5€-VPS in 15 Minuten aufgesetzt — und du bist nicht mehr von öffentlichen Servern abhängig.

Warum überhaupt selbst hosten?

  • Latenz — die offiziellen Public-Relays sitzen verteilt, aber nicht immer nah. Eigener Server in deinem Rechenzentrum = wenige ms.
  • Datenschutz — bei Relay-Verbindungen läuft dein Bildschirm-Stream durch fremde Server. Verschlüsselt, ja — aber metadata leakt.
  • Verfügbarkeit — die Public-Server sind manchmal überlastet, dein VPS ist verlässlich für dich allein.
  • Whitelist-Setup — wenn nur dein Server bekannt ist, ist deine ID auch nur dort auflösbar.

Was du brauchst

  • Ein VPS mit öffentlicher IPv4 — Hetzner CX11 (~4€/Monat), Netcup, OVH, DigitalOcean, alles tut. ARM oder x86, egal.
  • Bandbreite: hbbs braucht fast nichts (nur ID-Lookups). hbbr nur, wenn Verbindungen gerelayt werden — das passiert nur, wenn beide Peers hinter strikten NATs sind. Faustregel: für 5–10 User reichen 100 Mbit.
  • Optional: eigene (Sub-)Domain, z.B. rd.example.com → A-Record auf die VPS-IP.

Setup mit Docker (empfohlen)

Compose-File

# /opt/rustdesk/docker-compose.yml
services:
  hbbs:
    image: rustdesk/rustdesk-server:latest
    container_name: hbbs
    command: hbbs -r rd.example.com:21117
    volumes:
      - ./data:/root
    network_mode: host
    restart: unless-stopped

  hbbr:
    image: rustdesk/rustdesk-server:latest
    container_name: hbbr
    command: hbbr
    volumes:
      - ./data:/root
    network_mode: host
    restart: unless-stopped

Erklärung der Tricks:

  • network_mode: host — RustDesk braucht UDP-Hole-Punching, was mit Docker-NAT zu Headaches führt. Host-Mode umgeht das.
  • -r rd.example.com:21117 — sagt hbbs, wo der hbbr-Relay erreichbar ist. Wenn nur eine Maschine: derselbe Hostname, Port 21117.
  • Volume ./data — hier landen die Public-Keys. Nicht löschen, sonst ändern sich die Keys und alle Clients zeigen „Mismatch".
cd /opt/rustdesk
docker compose up -d
docker compose logs -f hbbs   # zeigt Public-Key beim ersten Start

Public-Key holen

cat /opt/rustdesk/data/id_ed25519.pub

Den 44-Zeichen-String brauchst du gleich auf den Clients.

Setup ohne Docker

Falls du Docker nicht magst:

cd /opt
wget https://github.com/rustdesk/rustdesk-server/releases/latest/download/rustdesk-server-linux-amd64.zip
unzip rustdesk-server-linux-amd64.zip

# systemd unit für hbbs
sudo tee /etc/systemd/system/hbbs.service <<EOF
[Unit]
Description=RustDesk hbbs
After=network.target

[Service]
WorkingDirectory=/opt/rustdesk
ExecStart=/opt/rustdesk/hbbs -r rd.example.com:21117
Restart=always

[Install]
WantedBy=multi-user.target
EOF

# analog für hbbr (ohne -r-Flag)
sudo systemctl enable --now hbbs hbbr

Ports & Firewall

PortProtokollWofür
21115TCPNAT-Test (optional, aber empfohlen)
21116TCPhbbs ID-Registrierung
21116UDPhbbs Heartbeats / Hole-Punch unbedingt
21117TCPhbbr Relay
21118TCPhbbs Web-Client (optional, Pro-Feature)
21119TCPhbbr Web-Client (optional)
 
# UFW
sudo ufw allow 21115:21119/tcp
sudo ufw allow 21116/udp

# nftables / iptables — analog freischalten
⚠️ Häufiger Fehler UDP 21116 vergessen → Direkt-Verbindungen funktionieren scheinbar, sind aber langsam, weil sie über Relay laufen. Mit allen Ports offen klappt P2P, ohne Relay-Last für deinen VPS.

Client auf den eigenen Server zeigen

Auf jedem Client (Windows, Mac, Linux, Android, iOS):

  1. Settings → Network → ID/Relay Server.
  2. ID Server: rd.example.com (Port wird automatisch 21116 sein).
  3. Relay Server: leer lassen — hbbs verteilt das automatisch via -r-Flag.
  4. API Server: leer (außer du hast Pro mit Web-Console).
  5. Key: der 44-Zeichen-String aus id_ed25519.pub.
  6. Apply → der ID-Wert ändert sich (neuer Server = neue ID).

Public Key — der Trick gegen MITM

Wenn du den Key nicht hinterlegst, akzeptiert der Client den ersten Public Key, den der Server schickt — Trust on first use. Wer den Server hijacken kann (DNS/Routing-Angriff), kann seinen eigenen Key unterschieben.

Mit hinterlegtem Key wird jede Server-Antwort gegen den Public Key validiert. Manipulation = sichtbare Warnung, Verbindung abgelehnt.

💡 Distribution-Trick Statt jedem Familienmitglied per Telefon den Key zu erklären: bau einen Custom Client mit hardcodierten Server-Settings. Eine Datei verschicken → Doppelklick → fertig. Mehr dazu auf der Profi-Seite.

Pro-Edition: lohnt sich das?

RustDesk-Server-Pro (kommerzielle Lizenz) bietet zusätzlich:

  • Web-Konsole — User/Permissions-Management im Browser
  • Address-Book-Sync — gemeinsames Adressbuch für ganzes Team
  • 2FA server-seitig erzwingbar
  • OIDC/LDAP-Login
  • Audit-Logs
  • Geo-Failover für mehrere Relays

Lizenz: ~$10/Monat aufwärts, je nach User-Zahl. Faustregel: bis 3 User reicht OSS komplett. Für ein kleines Unternehmen mit 10+ Geräten ist Pro die Investition wert.

Alternative für Pro-ähnliche Features ohne Bezahlmodell: rustdesk-api-server (Community-Projekt, ungesupportet, aber funktional).
