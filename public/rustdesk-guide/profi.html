1. Custom Client mit hardcodierter Config

Der mächtigste Trick im RustDesk-Universum. Du baust einen Installer, der bereits weiß, wo dein Server steht und welcher Public-Key gilt — und schickst ihn an Familie/Kunden. Die machen Doppelklick und sind verbunden, ohne irgendwas zu konfigurieren.

Der einfache Weg: Rename-Trick

RustDesk parst seinen eigenen Dateinamen beim Start. Wenn du den Installer nach einem bestimmten Schema benennst, übernimmt er die Settings beim ersten Start.

  1. In der App: Settings → Network → fülle ID-Server, Key etc. aus.
  2. Settings → "..." (oben rechts) → Export Server Config → du bekommst einen Base64-String.
  3. Lade den normalen Installer.
  4. Benenn ihn um nach dem Schema: 
    rustdesk-host=<hbbs-host>,key=<public-key>.exe

    Beispiele:

    rustdesk-host=rd.example.com,key=ABC...XYZ.exe
  5. Empfänger lädt das Ding herunter, doppelklickt → installiert mit deinen Settings vorausgefüllt.

Der saubere Weg: api.rustdesk.com

Auf api.rustdesk.com (offizieller Build-Service) gibst du deine Server-Adresse + Key + ein paar Tweaks ein, klickst „Build", lädst nach 2–3 Minuten einen vollständig vorkonfigurierten Installer (Windows/Mac/Linux). Kostet ein bisschen, dafür sauber signiert.

Der DIY-Weg: selbst kompilieren

Repo clonen, in libs/hbb_common/src/config.rs die Default-Werte für RENDEZVOUS_SERVERS, RS_PUB_KEY etc. ändern → kompilieren. Detail-Anleitung steht im RustDesk-Wiki unter „Customize the Software".

💡 Why this matters Familie + Freunde betreuen ist eine Sache von Sekunden. „Klick auf diesen Link, klick auf das EXE, fertig." Keine Telefon-Anleitungen, keine Server-Adresse-tippen, keine Public-Key-Mismatch-Verwirrung.

2. TCP-Tunneling — VPN-Ersatz für einen einzelnen Port

RustDesk kann beliebige TCP-Ports über die Verbindung tunneln. Bedeutet: du kannst von deinem Laptop aus auf einen Service hinter NAT zugreifen, ohne VPN aufzubauen.

Beispiel: SSH zum Heim-Server

  1. Auf dem Heim-PC läuft RustDesk-Service. SSH-Server lauscht lokal auf Port 22.
  2. Vom Laptop verbindest du dich auf den Heim-PC (Permanent Password).
  3. In der Verbindungs-Toolbar: TCP Tunneling → "+ Add Tunnel".
  4. Local Port: 2222, Remote Host: localhost, Remote Port: 22.
  5. Auf dem Laptop: ssh -p 2222 user@localhost → bist auf dem Heim-PC.

Funktioniert für alles TCP: VNC, RDP zu einem anderen Rechner im Heimnetz, Web-UIs (Synology, Pi-hole, Home Assistant), Datenbank-Verbindungen.

Use-Case Heim-Server hat ein Home-Assistant-Webinterface auf http://homeserver.lan:8123. Du tunnelst local 8123 → homeserver.lan:8123 über deinen anderen Heim-PC und öffnest http://localhost:8123 auf dem Laptop. Wireguard-frei.

3. Privacy Mode (Display-Blackout)

Wenn du remote auf einem Office-PC arbeitest, sollte niemand vor Ort sehen, was du tust. Privacy Mode macht den lokalen Bildschirm schwarz während der Session.

Aktivierung: in der Toolbar während Verbindung → 🔒 Privacy Mode → On.

Voraussetzungen

  • Windows-Host (auf macOS/Linux nicht verfügbar — OS-Limit).
  • RustDesk Privacy Mode Driver installiert (Settings → Display → "Install Privacy Mode Driver"). Es ist ein virtueller Display-Treiber, der einen schwarzen Bildschirm vorgaukelt.
  • Auf neueren Builds gibt's alternativ "virtual display" Modus, der ohne Treiber-Install auskommt — funktioniert aber nicht auf jeder GPU.

Lokale Tastatur/Maus werden ebenfalls geblockt — wenn jemand am Schreibtisch wackelt, passiert nichts.

4. Codecs & Hardware-Acceleration

Default-Codec ist VP9. Funktioniert immer, ist aber CPU-lastig. Wenn dein PC eine moderne GPU hat, kannst du auf Hardware-Codecs umstellen für deutlich flüssigere Bilder bei niedrigerer CPU-Last.

In der Toolbar während Verbindung → Display → Codec:

CodecWann
VP9Default. Software-encoded. Verlässlich, mittlere Performance.
VP8Älterer Codec. Wenn VP9 stottert, manchmal stabiler. Größere Bandbreite.
AV1Modernster Codec. Beste Qualität bei niedriger Bandbreite. CPU-hungrig auf Hosts ohne AV1-Hardware-Encoder. Auf neuen Intel/AMD/Nvidia GPUs der Codec der Wahl.
H264Hardware-Acceleration auf praktisch jedem Rechner. Sehr schnell. Etwas größere Files. guter Default für Gaming/Video
H265Wie H264, bessere Qualität pro Bit. Erfordert HEVC-Hardware-Encoder.

Settings → Display → "Use hardware codec" einschalten + Codec wählen. Beim Verbinden zu einem Linux-Host mit vaapi-fähiger GPU funktioniert H264-HW-Encode auch dort.

💡 Quality vs. Speed Im Verbindungs-Toolbar gibt's einen Image-Quality-Slider: Best, Balanced, Low, Custom. „Custom" lässt dich Bitrate und Quantizer manuell setzen — Sweet-Spot für DSL-Leitungen meistens 2–3 Mbit, FPS auf 30 begrenzt. Bei mobilem 4G: 1 Mbit, 20 FPS.

5. Wake-on-LAN über einen Always-On-Peer

Klassisches Problem: Heim-PC ist aus, du willst dich verbinden. Lösung: ein winziger Always-On-Peer im Heimnetz (Raspberry Pi 4 reicht, oder dein NAS) sendet das WoL-Magic-Packet.

Setup

  1. Auf dem Heim-PC im BIOS Wake-on-LAN aktivieren. Im OS: Netzwerkadapter → "Wake on Magic Packet".
  2. Auf dem Always-On-Peer (RPi mit Linux): RustDesk-Service installieren, Permanent Password setzen.
  3. Pro-Trick: in RustDesk-Settings auf dem RPi einen Custom-Befehl hinterlegen, der wakeonlan AA:BB:CC:DD:EE:FF ausführt — kann via Run program-Permission ferngesteuert werden.
  4. Vom Laptop: RustDesk → connect zum RPi → Toolbar → "Run command" → WoL feuert ab.
  5. 30 Sekunden warten → Verbindung zum Heim-PC.

Alternative: RustDesk-Pro hat WoL als nativen Toolbar-Button im Address Book. Kein Pi nötig, aber Pro-Lizenz.

6. CLI-Tricks & Scripting

Der RustDesk-Client hat einen unterdokumentierten CLI-Modus:

# Direkt verbinden mit Passwort (Linux/Mac)
rustdesk --connect 123456789 --password "deinpasswort"

# File-Transfer-Modus direkt öffnen
rustdesk --file-transfer 123456789 --password "..."

# Port-Forwarding nur (kein Bildschirm)
rustdesk --port-forward 123456789

# Config-Datei lesbar machen
cat ~/.config/rustdesk/RustDesk.toml

Use-Case: tägliches Connect-Skript zur Arbeitsmaschine — alias work auf der Shell, das die ID + Pass aus deinem Passwort-Manager zieht und direkt connectet.

Headless-Server steuern

Für Linux-Server ohne Display brauchst du einen virtuellen X-Server (Xvfb). RustDesk-Wiki hat dazu eine Anleitung — funktioniert, ist aber Frickel-Arbeit. Für Server-Admin sind SSH + tmux meistens die bessere Wahl.

7. IP-Allowlist & Whitelist-IDs

Settings → Security:

  • Whitelist (IP) — nur Verbindungen aus diesen IPs/CIDRs werden überhaupt akzeptiert. Format: 192.168.1.0/24, 84.144.0.0/16. Hochwirksam gegen Brute-Force von außen.
  • Verifiable Public-IDs (Pro) — nur Clients, die auf einer Liste stehen, dürfen verbinden. Selbst mit korrektem Passwort: kein Connect, wenn ID nicht gelistet.

Kombinier IP-Allowlist mit dynamischer DNS: ein Cron-Job auf dem Heim-PC schreibt deine aktuelle Office-IP regelmäßig in die Allowlist. So bist du auch bei dynamischen IPs immer erreichbar, aber nicht für die Welt.

8. Multi-Monitor sauber handhaben

Wenn der Host mehrere Monitore hat, hast du in der Toolbar eine Display-Auswahl:

  • Single-Monitor — nur ein Display, nimm das aktive.
  • All Displays — alle nebeneinander in einem großen Stream. Bandbreiten-hungrig, aber praktisch.
  • Mehrere Fenster — pro Display ein eigenes RustDesk-Fenster (Pro-Feature in einigen Builds).

Auf modernen Builds gibt's die Option Show remote cursor + Track active display. Wenn Track-active an ist, springt der Stream automatisch zum Display, auf dem die Maus gerade ist. Praktisch, manchmal verwirrend — kennst du den Stream-Wechsel nicht, denkst du, der Bildschirm „spinnt".

9. Mobile-spezifische Tweaks

  • Datensparmodus: in der Verbindung → Display → Image quality → "Low". Spart bei mobilem Internet ~70% Bandbreite, schlechte Lesbarkeit.
  • Show remote cursor: aus auf dem Telefon → Cursor-Position kommt aus dem letzten Tap, nicht dauernd übertragen → spart minimal Bandbreite, fühlt sich aber „kaputter" an. Default an lassen.
  • Bildschirm-aus-Modus: bei Android — wenn der Screen schwarz wird, läuft die Verbindung weiter (Background-Service-Berechtigung vorausgesetzt).
  • Floating Action Button auf Android — kannst du verstecken, wenn er stört. Settings in der Verbindung selbst.

10. Wirklich ungewöhnliche Use-Cases

RustDesk als „Kiosk-Display"

Eine Steam-Box / Nuc / RPi an einem Fernseher. Auf dem Couch-Laptop läuft RustDesk im Vollbild → quasi „dein Laptop sendet auf den TV". Latenz mit H264 + Direct-Connect: 50–80ms — gut für Filme, mäßig für schnelle Spiele, aber für Office-Demo top.

Eltern-Support, ohne dass sie was tun müssen

Custom Build mit hardcodiertem Server + permanentem Passwort + Service-Install + IP-Allowlist auf deine IP. Eltern kriegen einmal den Installer per Mail, Doppelklick — du kannst danach jederzeit ohne ihre Aktion ran. Dazu in den Permissions require local approval AUS — sonst müssen sie doch klicken.

Drei-Klick-Onboarding für externe Mitarbeiter

Custom Build inkl. vorbefülltem Permanenten Passwort (geht via Config-Override). Mitarbeiter installiert, du hast sofort Zugriff auf den Office-PC. Combine with Service-Account-Tokens auf API-Server-Seite für Audit-Logs.

RustDesk + Tailscale = Overkill, aber wasserdicht

Wenn du eh Tailscale im Netz hast: RustDesk-Server auf einer Tailscale-IP betreiben. ID-Server-Adresse zeigt auf 100.x.y.z. Außerhalb des Tailscale-Netzes ist dein RustDesk-Setup komplett unsichtbar — Public-Key + Tailscale-WireGuard. Maximale Paranoia, minimal Reibung.

Audio-Streaming als „Spotify-Speaker"

Audio-Permission an, dann hörst du den System-Sound des Hosts. Klingt nach Trivia, aber: dein Heim-Server spielt FLAC ab, du hörst es vom Laptop. Latenz ist nicht optimal für Live-Hören, aber okay für Hintergrundbeschallung.
