1. Wie RustDesk funktioniert (in 60 Sekunden)

Drei Komponenten arbeiten zusammen:

  • hbbs (ID/Rendezvous-Server) — Telefonbuch. Speichert „ID 123 456 789 ist gerade an dieser IP erreichbar". Kein Datenverkehr läuft hier durch.
  • hbbr (Relay-Server) — Notausgang. Wenn beide Peers hinter strikten NATs sitzen und kein Direkt-Connect klappt, läuft der Traffic verschlüsselt durch hier durch.
  • Client — die App, die du installierst. Sowohl der Steuernde als auch der Gesteuerte nutzen dieselbe App.

Standard: RustDesk benutzt die öffentlichen Server von rustdesk.com. Du kannst stattdessen eigene betreiben — das ist der Hauptgrund, warum man RustDesk nutzt.

2. Installation pro Plattform

Windows

Download von rustdesk.com. Beim Start hast du drei Optionen:

  • Run — Portable. App läuft, aber nur solange du eingeloggt bist und sie offen ist. Gut zum Ausprobieren.
  • Install — Volle Installation, läuft als Windows-Dienst. Pflicht für Unattended Access.
  • Install via system service — wie oben, plus UAC-Bypass: du kannst remote auf Admin-Prompts klicken.
💡 Wichtig Wenn du remote Software installieren oder UAC-Dialoge wegklicken willst, muss RustDesk als Service laufen. Sonst wird der UAC-Dialog dunkel angezeigt und Eingaben kommen nicht durch (Windows-Sicherheitsfeature).

macOS

Download → in /Applications ziehen. Beim ersten Start drei Permissions setzen:

  1. Bildschirmaufnahme (Screen Recording)
  2. Bedienungshilfen (Accessibility)
  3. Eingabeüberwachung (Input Monitoring)

Wer eine vergisst → schwarzes Bild oder Maus reagiert nicht. Nach dem Erteilen die App komplett beenden und neu starten (sonst greifen die Permissions nicht).

Linux

.deb, .rpm oder AppImage. Wayland ist noch immer zickig — auf Wayland-Sessions geht oft nur View, kein Input. Workaround: zur X11-Session loggen.

# Debian/Ubuntu
sudo apt install ./rustdesk-1.x.x.deb

# Service starten (für Unattended)
sudo systemctl enable --now rustdesk

Android

Aus dem Play Store oder als APK von github.com/rustdesk/rustdesk (offizielle Releases). Es gibt zwei Apps:

  • RustDesk — die normale App. Genug, um Android zu steuern.
  • RustDesk Input — separate Sub-App. Pflicht, wenn jemand vom PC aus dein Android steuern soll. Liefert die Input-Methode, da Android das aus Sicherheitsgründen nicht in einer normalen App erlaubt.

iOS

Im App Store. iOS lässt sich aus Apple-Restriktionen nicht fernsteuern — du kannst von iOS aus nur steuernd agieren, nicht gesteuert werden.

3. Die 9-stellige ID

Beim ersten Start würfelt RustDesk eine 9-stellige ID. Die ist an dein Gerät + die hbbs-Konfiguration gebunden. Wichtig:

  • IDs werden in ~/.config/rustdesk/RustDesk.toml (Linux/Mac) bzw. %APPDATA%\RustDesk\config\RustDesk.toml gespeichert.
  • Wenn du dieselbe Config kopierst, kriegst du dieselbe ID — nützlich zum Migrieren.
  • ID ändern: in der App auf das ✏️-Symbol neben der ID. Geht nur einmal alle paar Stunden (Anti-Spam).
  • IDs sind pro Server. Wechselst du den Rendezvous-Server, bekommst du eine neue ID.

4. One-Time vs. Permanent Password

Unter dem ID-Feld siehst du ein Passwort-Feld mit einem 🔄-Symbol. Das ist der Kern der Sicherheit.

ModusWann
One-Time (Default)6-stelliger Code, ändert sich bei jedem Verbinden. Optimal für „mein Vater ruft an, ich helfe einmal". Code per Telefon ansagen.
PermanentDauerhaftes Passwort. Pflicht für Unattended Access. Setz es in Settings → Security → Permanent Password.
Beides aktivDefault-Verhalten: dein permanentes Passwort funktioniert, aber jemand kann sich auch mit One-Time-Code verbinden, wenn du den ansagst.
⚠️ Permanent Password ≠ schwaches Passwort Wenn deine ID bekannt ist und das Permanent Password schwach, kann jeder Brute-Force versuchen. Mindestens 12 Zeichen, generiert aus deinem Passwort-Manager. RustDesk hat eingebaute Brute-Force-Sperre, aber auf das eigene Setup achten.

5. Erste Verbindung Schritt-für-Schritt

  1. Auf Gerät A (das gesteuert wird): RustDesk öffnen. ID + 6-stelliger Code stehen prominent oben.
  2. Auf Gerät B (das steuert): RustDesk öffnen. Im großen Eingabefeld die ID von A eintippen → "Connect".
  3. A bekommt einen Permission-Dialog: „User XYZ möchte sich verbinden, View / Control / File Transfer …"
  4. Code von A in B eingeben → Verbindung steht.

Beim ersten Connect wird ein Public-Key-Fingerprint ausgetauscht („Trust on first use"). Bei späteren Verbindungen zur selben ID wird der Fingerprint geprüft. Ändert er sich → dicke Warnung („Key Mismatch", siehe Troubleshooting).

6. Permissions — was die Gegenseite darf

Pro Verbindung kann die gesteuerte Seite einzeln freigeben:

Keyboard / MouseEingabe-Steuerung. Aus = nur View-Modus.
ClipboardCopy-Paste zwischen Geräten. Praktisch, aber wer paranoid ist: aus.
File transferDrag-and-Drop von Dateien.
AudioSystem-Sound vom gesteuerten zum steuernden Gerät.
TCP tunnelingPort-Forwarding über die Verbindung. Für Profis (siehe Profi-Tipps).
RecordingSitzung mitschneiden — wer das tut, sieht der andere.
RestartRemote-Reboot des Geräts.
Block user inputLokale Maus/Tastatur sperren während Session — Standard-Pranger-Feature.

Setz die Defaults global unter Settings → Security → Permissions. Pro Verbindung kannst du sie weiter eingrenzen.

7. Unattended Access einrichten

„Unattended" = du steuerst ein Gerät, an dem niemand sitzt, der den Code ansagen könnte. Klassisch: dein Heim-PC vom Office aus.

  1. Auf dem Zielgerät RustDesk als Service installieren (Windows: "Install via system service"; Linux: systemctl enable --now rustdesk).
  2. Settings → Security → Permanent Password setzen.
  3. Optional: Approve mode auf password stellen (statt password and click) → keine manuelle Bestätigung am Zielgerät nötig.
  4. ID + Permanent Password im Address Book auf dem steuernden Gerät speichern (Sidebar → Address Book → "+").
  5. Test-Connect, dann Reboot, dann nochmal Test-Connect — der zweite Test ist der entscheidende.
💡 Pro-Move für Unattended-Setups Setz zusätzlich eine IP-Allowlist (Settings → Security → IP Allowlist) — nur Verbindungen von deinen bekannten IPs werden überhaupt akzeptiert. Brute-Force-resistent, weil ein Angreifer schon den Verbindungs-Versuch nicht aufbauen kann. Funktioniert auch, wenn du eine dynamische IP hast — ein CIDR-Bereich deines ISPs reicht.
