Verbindung „klemmt" auf falschem Zoom-Level

Symptom: Du steuerst Android (oder einen anderen Host) vom Windows-PC und alles ist herangezoomt. Scrollen, Maus-Wackeln, Resize ändert nichts.

Das passiert in fast allen Fällen, weil View Style oder Image Ratio auf einem Wert hängen, den du nicht erwartest. Reset:

  1. In der oberen Toolbar der Verbindung auf Display (Bildschirm-Symbol) klicken.
  2. Im Menü View Style:
    • Original = 1:1 Pixel-Mapping. Wenn das Remote-Display kleiner als dein lokales Fenster ist, wird's aussehen wie zentriert. Wenn es größer ist, musst du scrollen.
    • Shrink = Bild wird in dein Fenster eingepasst (proportional). der „normale" Modus
    • Stretch = Bild wird gedehnt, ggf. verzerrt.
  3. Im selben Menü Image Quality → Original oder Adaptive.
  4. Im selben Menü Scroll Style auf Auto.
  5. Wenn du immer noch klebst: Verbindung trennen, Toolbar oben rechts → "..." → Reset Canvas (nicht in jedem Build da, aber in 1.3+).

Der häufigste Auslöser: versehentlicher Zoom per Ctrl+Scroll

RustDesk hat einen versteckten Zoom-Modus: Ctrl + Mausrad scrollt den Stream rein/raus. Wenn du im View-Stil "Original" bist und gescrollt hast, bleibt der Zoom — Resize hilft nicht, weil "Original" eben kein Auto-Fit macht.

Fix in einem Schritt: View Style auf Shrink setzen → Bild snappt zurück auf „in Fenster eingepasst". Dann optional zurück auf Original.

💡 Konkret für Win → Android Bei Verbindungen zum Telefon ist Shrink fast immer richtig — sonst hast du ein winziges Telefon-Display 1:1 in einem riesigen Fenster, oder ein hochskaliertes pixeliges Bild. Setz das zur Default-Einstellung in Settings → Display → Default View Style, dann hast du das Problem nie wieder.

Schwarzer Bildschirm

Verbindung steht (Toolbar reagiert), aber das Bild ist schwarz.

PlattformUrsache & Fix
macOSScreen-Recording-Permission nicht gesetzt oder nicht griffig (App neu starten nach Permission). Systemeinstellungen → Datenschutz & Sicherheit → Bildschirmaufnahme.
WindowsPrivacy Mode ist versehentlich an (Toolbar → 🔒 prüfen). Oder: Hardware-Codec inkompatibel, in Settings auf Software/VP9 zurück.
AndroidMediaProjection nicht erteilt — RustDesk öffnen, „Start Service" tippen, Dialog erlauben. Oder: DRM-Inhalt im Vordergrund (Netflix etc).
Linux/WaylandWayland blockt Screen-Capture. Workaround: zur Xorg-Session loggen, oder PipeWire-portal-fähigen Build nutzen (RustDesk 1.3+ unterstützt das experimentell).

Tastatur-Layout ist falsch

Symptom: Du tippst „ä", Remote schreibt „;". Tippst „y", Remote bekommt „z".

RustDesk hat drei Keyboard-Modi (in der Verbindungs-Toolbar → Tastatur-Symbol):

  • Legacy — Roh-Keystrokes 1:1. Funktioniert, wenn beide Seiten dasselbe Layout haben. Bei Mismatch (US ↔ DE) kommt Müll raus.
  • Map — RustDesk übersetzt von deinem lokalen Layout auf das Remote-Layout. 90% der Fälle Hier wird's richtig.
  • Translate — schickt Unicode-Zeichen statt Keystrokes. Macht Spiele kaputt (Gaming braucht WASD-Keystrokes), aber für Sondertastenkombinationen / IME-Eingabe (Chinesisch, Japanisch) Pflicht.

Default ist Map. Wenn das Layout falsch ist, wechsle die Modi durch — meistens fixt einer der anderen es.

„Key Mismatch" Warnung

Beim Connect kommt ein roter Banner: „This server's key has changed".

  1. Self-Hosted Server wurde neu deployed → id_ed25519-Datei wurde neu generiert. Public Key auf allen Clients aktualisieren oder Server-Datenverzeichnis aus Backup wiederherstellen.
  2. MITM-Angriff — extrem selten, aber bei kritischen Verbindungen ernst nehmen. Hörmal beim Server-Admin per anderem Kanal nach, ob der Key wirklich gewechselt hat.
  3. Du hast versehentlich auf einen anderen Server umgestellt → Settings → Network checken.

Service stoppt nach Reboot (Android)

Auf Xiaomi/Samsung/OnePlus killt das Akku-Optimierungs-Framework Background-Services. Drei Schritte:

  1. Android-Settings → Apps → RustDesk → Akku → "Nicht eingeschränkt" / "Keine Optimierung".
  2. Dasselbe für RustDesk Input.
  3. Auf Xiaomi: Sicherheits-App → "Berechtigungen" → "Autostart" → RustDesk + Input erlauben.

Wenn das nicht reicht: ein Tasker-Macro „On Boot Completed → Open RustDesk → Tap Start Service". Hässlich, aber funktioniert.

„Cannot control" auf Android trotz Input-App

RustDesk Input ist installiert, du siehst das Display, aber Maus-Bewegungen kommen nicht durch.

  1. Android-Settings → Bedienungshilfen (Accessibility) → RustDesk Input → eingeschaltet?
  2. Beim Aktivieren wird Android dich warnen, dass die App „den Bildschirm beobachtet" — das ist normal und nötig.
  3. Auf Android 14+ kommt noch die Permission „Display over other apps" für RustDesk Input dazu.
  4. Nach dem Toggle: Verbindung beenden und neu aufbauen.

Ruckliges Bild / hohe Latenz

Checkliste, ungefähr in der Reihenfolge der Häufigkeit:

  1. Connection-Type prüfen: Toolbar → Verbindung-Info. Steht da "P2P / Direct" → gut. Steht da "Relay" → Traffic läuft über Server, Latenz +. Lösung: eigenen Server oder Port-Forwarding daheim für UDP 21116.
  2. Codec wechseln: bei guter GPU auf H264 oder AV1, bei schwacher auf VP9.
  3. Bitrate runter: Image Quality → "Custom" → 2 Mbit, 30 FPS. Sieht weniger schick aus, ist aber stabil.
  4. FPS-Lock: 30 statt 60 reicht meistens, halbe Bandbreite.
  5. Hardware-Encoder: in Settings einschalten. Verlagert Encode-Last von CPU auf GPU.

UAC-Dialog ist dunkel / nimmt keine Eingaben

Klassisches Windows-Verhalten. Lösung:

  • RustDesk muss als Service laufen (Settings → "Install via system service" oder per CMD rustdesk --install-service).
  • Während der Session in der Toolbar: Elevate klicken → RustDesk hebt sich selbst auf System-Level.
  • Erst dann werden UAC-Dialoge sichtbar und steuerbar.

Linux / Wayland

Wayland ist immer noch das Sorgenkind. Status:

  • RustDesk 1.3+ kann via PipeWire-Portal Screen-Capture machen — der Browser-artige „Welches Fenster soll geteilt werden"-Dialog.
  • Input-Capture (also Maus/Tastatur ans System schicken) geht auf Wayland nur, wenn der Compositor `libei` unterstützt — KDE 6 ja, GNOME 46+ teilweise.
  • Workaround: GDM/SDDM auf X11 zurückstellen (auf den meisten Distros: Login-Screen → Zahnrad → "Xorg" auswählen).

„Not reachable" / verbindet nicht

  1. Beide Geräte online und im Internet?
  2. Auf beiden derselbe ID-Server eingestellt? (Settings → Network)
  3. Firewall lässt RustDesk raus?
  4. Self-Hosted: nc -vz rd.example.com 21116 vom Client erreichbar? UDP auch?
  5. VPN am Client aktiv und blockiert UDP? Probier mal ohne VPN.
  6. Ein Reboot beider Geräte ist tatsächlich oft die Lösung — der hbbs-Eintrag verfällt nach Idle und der Client meldet sich neu an.
