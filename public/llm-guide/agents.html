Tools & Agents

Ein nacktes LLM kann Texte schreiben. Mit Tools kann es im Web suchen, Code ausführen, APIs aufrufen, Datenbanken abfragen, Termine buchen. Dieselben Modelle — andere Hebel.

Tool-Use, Schritt für Schritt

  1. Du registrierst beim Modell ein Tool: Name, Beschreibung, Parameter-Schema (JSON Schema).
  2. User stellt Frage. Modell entscheidet: brauche ich ein Tool?
  3. Falls ja: Modell gibt einen Tool-Call aus — Strukturiertes JSON mit Tool-Name + Argumenten. Es führt das Tool nicht selbst aus.
  4. Dein Code ruft das Tool auf, kriegt das Ergebnis, gibt's ans Modell zurück.
  5. Modell verwertet das Ergebnis und antwortet — oder ruft das nächste Tool.

Wichtig zu kapieren: Modell ruft NIEMALS direkt eine API. Modell schlägt vor, du führst aus. Sicherheits-Geländer steht damit zwischen Modell und Welt.

Beispiel — Wetter-Tool

Tool-Definition:
{
  "name": "get_weather",
  "description": "Aktuelles Wetter für eine Stadt.",
  "parameters": {
    "type": "object",
    "properties": {
      "city": { "type": "string" }
    },
    "required": ["city"]
  }
}

User: Wie ist das Wetter in Berlin?

Modell-Antwort (Tool-Call):
{
  "tool": "get_weather",
  "arguments": { "city": "Berlin" }
}

Dein Code: API aufrufen → { "temp": 12, "condition": "Regen" }

Du gibst zurück ans Modell:
{ "result": { "temp": 12, "condition": "Regen" } }

Modell-Antwort (final):
"In Berlin ist es gerade 12°C und es regnet."

Standard-Werkzeuge auf der Baustelle

  • Web Search — Bing / Tavily / Brave / Perplexity-API
  • Code Interpreter — Python in Sandbox ausführen, Plots erzeugen, Daten analysieren
  • File Reader — PDFs, Excel, CSV laden und mitgeben
  • SQL-Query — kontrollierter DB-Zugriff (read-only empfohlen)
  • Calendar / Email — typische Office-Integrationen
  • Browser-Automation — Playwright/Computer-Use für Klicken im Browser

Vom Tool-Use zum Agenten

Ein Agent ist ein Modell, das in einer Schleife läuft: denken → Tool aufrufen → Ergebnis verarbeiten → entscheiden, ob fertig oder weiter. Bis Stop-Bedingung. Architektur:

while not done:
    response = llm.invoke(messages, tools=tools)
    if response.is_tool_call:
        result = run_tool(response.tool_call)
        messages.append(result)
    else:
        done = True
        return response.text

Der Witz: das Modell entscheidet selbst, wann es fertig ist. Das ist Macht. Das ist Risiko.

Sicherheits-Vorschriften (lies mal richtig)

RisikoMaßnahme
Endlos-LoopMax-Iterations setzen (10–50)
Destruktive Tool-Calls (DELETE, DROP, send_email)Confirm-Step durch Mensch oder Allowlist
Prompt Injection via Tool-ResultTool-Outputs sanitizen, keine Markdown-Links blind anklicken
Datenleck durch unkontrollierte Web-SearchAllowlist von Domains, kein Auto-Fetch
Kosten-Explosion durch LoopsToken-Limits, Cost-Limits pro Run
⚠️ Vorarbeiter-Regel #1 JEDES Tool, das was außerhalb des Sandboxes verändert (Mail senden, Geld bewegen, Daten löschen, Code mergen), bekommt einen Confirm-Schritt. Modell schlägt vor — Mensch klickt OK. Wer das überspringt, kriegt Geschichten zu erzählen, die niemand hören will.

Multi-Agent-Setups

Mehrere spezialisierte Agenten, die zusammenarbeiten. Beliebte Pattern:

  • Planner + Executor — einer plant, einer arbeitet ab.
  • Spezialisten-Pool — Researcher, Coder, Reviewer, Editor.
  • Critic — separater Agent prüft Output des ersten.

Ehrlich gesagt: in 80% der Fälle reicht ein einzelner Agent mit guten Tools. Multi-Agent ist hipp, oft over-engineered. Erst Komplexität rechtfertigen, dann bauen.
